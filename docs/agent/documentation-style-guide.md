# 文档编写范式

这份文档定义 create-yume 当前的文档分流规则，用来避免把不同读者写进同一套叙述里。

## 1. 目标

文档体系现在分为两类：

- 用户文档：给人看
- 执行文档：给执行者看

两类文档都可以准确，但写法不能相同。

如果把它们混在一起，结果通常是：

- 用户觉得文档太硬、太内部化，不知道从哪里开始
- 执行者拿到的约束不够明确，容易偏移实现边界

## 2. 用户文档的写法

用户文档服务的是第一次接触项目的人、日常使用者和普通贡献者。

### 写作要求

- 使用简体中文
- 描述性表达优先
- 语气友好、清楚、可连续阅读
- 先说目标，再说步骤，再说边界
- 优先解释“这是什么”和“我该怎么做”

### 应包含的内容

- 项目用途
- 支持范围
- 常见操作入口
- 协作方式
- 使用时会碰到的主要边界

### 不应成为重点的内容

- 对执行者的强约束命令
- 过度内部化的实现细节
- 执行层面的检查清单
- 高密度流程控制语言

## 3. 执行文档的写法

执行文档服务的是在仓库里执行修改、验证和维护动作的执行者。

### 写作要求

- 命令式表达优先
- 明确约束、前提、检查点和禁止项
- 先给边界，再给流程，再给验证
- 目标是降低偏移，而不是提升阅读氛围
- 可以直接写“必须”“不要”“先做什么，再做什么”

### 应包含的内容

- 修改边界
- 高风险区域
- 推荐执行顺序
- 真实能力边界
- 最低验证要求
- 与执行正确性直接相关的约束

### 不应成为重点的内容

- 面向外部读者的项目介绍
- 冗长背景故事
- 友好但无约束力的泛化描述

## 4. 分流规则

判断一份文档应该放在哪里，只问一个问题：

**读者读完后要做什么？**

- 如果读完后是“使用、理解、接入、协作”，放到 `user`。
- 如果读完后是“修改、约束执行、验证、避免偏移”，放到 `agent`。

同一主题若同时服务两类读者，应拆分成两份文档，而不是在一页里混写两种口吻。

## 5. 根 README 的职责

根 README 是用户入口。

它应该：

1. 用清楚的语言说明 create-yume 是什么。
2. 让新读者快速开始使用。
3. 把维护者和执行者引导到 `docs/agent/`，而不是在 README 中堆内部约束。

README 不应承担完整的执行操作手册职责。

## 6. 维护约定

1. 新建文档前，先确定读者类型。
2. 文档标题要反映用途，而不是作者视角。
3. 长期文档优先描述行为、职责和边界，避免依赖易过期的源码路径。
4. 新文档必须被入口页链接，不能成为孤岛。
5. 如果现有文档同时出现用户口吻和执行口吻，应优先拆分重写。

## 7. 执行检查清单

新增或重写文档时，至少检查以下问题：

- 这份文档写给谁？
- 读完后对方能做什么？
- 现在的语气与读者匹配吗？
- 是否把另一类读者才关心的内容混了进来？
- 是否已经从正确入口可以发现它？

如果这些问题不能明确回答，这份文档还不够稳定。
