# 文档约束

## 文档入口

1. 所有用于指向其他文档的引导文档，统一命名为 `roadmap.md`。
2. 根 `README.md` 是用户入口，不是执行操作手册。
3. 目录级 `README.md` 不再承担引导职责；目录引导统一收敛到 `roadmap.md`。
4. 新文档必须从至少一个 `roadmap.md` 或根 `README.md` 可被发现。

## 文档分流

1. 用户文档与执行文档必须分开写。
2. 服务“使用、理解、协作”的内容属于用户文档。
3. 服务“修改、约束执行、验证、避免偏移”的内容属于执行文档。
4. 同一主题如果同时服务两类内容，应拆成两份文档，而不是混写在一页里。

## 语言与表达

1. 说明性文字统一使用简体中文。
2. 技术术语、专业术语、命令、包名、文件名、代码标识符保留原始英文。
3. 用户文档优先描述性表达、友好语气和可读性。
4. 执行文档优先命令式表达、边界、流程、检查点和禁止项。

## 同步更新

1. 修改用户可见行为时，必须检查是否需要同步更新用户文档。
2. 修改执行边界、验证方式、模板能力或约束规则时，必须同步更新对应执行文档。
3. 修改文档入口结构时，必须同步检查所有 `roadmap.md` 与根 `README.md` 的链接。
4. 不允许让代码现状和文档现状长期分叉。

## 文档定位

1. `docs/` 只保留稳定、当前、描述性的文档。
2. `.gsd/PROJECT.md` 用于记录当前项目状态。
3. `.gsd/DECISIONS.md` 用于记录可持续决策。
4. `.gsd/REQUIREMENTS.md` 用于记录能力契约。
5. 历史流程残留、阶段性归档、旧审查材料不应再当作当前文档使用。
