# 当前 Handlebars Helper 说明

目标只有一个：**明确当前模板系统真实支持什么，不要按想象扩展能力边界。**

## 先确认一个事实

当前仓库**没有**接入 `handlebars-helpers` 这个第三方 helper 集。

因此，模板中可用的自定义 helper 只来自仓库内当前实现，不能假设常见 helper 默认存在。

## 当前可用的自定义 helper

目前只有 3 个：

| Helper | 作用 | 当前语义 |
| --- | --- | --- |
| `eq` | 严格相等判断 | 比较两个值是否严格相等 |
| `or` | 返回第一个 truthy 值 | 常用于条件判断或派生局部值 |
| `withHash` | 扩展当前块作用域 | 在保留父级上下文的前提下合并临时字段 |

## 使用时的判断原则

### `eq`

适合做明确分支判断。

当模板逻辑依赖配置项是否等于某个值时，优先使用它，而不是把比较逻辑拆散到更隐晦的表达式里。

### `or`

它不是“只返回布尔值”的 helper。

它返回第一个 truthy 操作数；如果前面的值都不成立，就返回最后一个值。写模板时要按这个语义理解，而不是把它当成简单布尔运算符。

### `withHash`

适合在块内部引入局部派生值。

当同一个条件会重复出现时，用 `withHash` 提升可读性，避免在一个模板块里反复展开长条件。

## 当前模板作者应遵守的边界

1. 不要在模板里使用仓库没有明确提供的自定义 helper。
2. 需要新 helper 时，先把能力边界设计清楚，再补实现、测试和文档。
3. 不要让模板依赖“也许存在”的隐式运行时能力。
4. 如果某个 helper 的语义不够直观，优先改写模板结构，不要继续堆表达式技巧。

## 发生变更时必须同步的事项

如果你新增或调整 helper，需要同步完成三件事：

1. 更新 helper 实现。
2. 补上或更新对应测试。
3. 回写这份文档，说明新的真实能力边界。

缺一不可。否则后续执行者会在错误假设上继续扩展模板。
