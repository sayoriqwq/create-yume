# pnpm Monorepo 约定

这份文档写给会在仓库里安装依赖、更新依赖和执行常规开发命令的人。

如果你读完后只需要完成一件事，那就是：**按仓库当前约定安全地管理依赖，并知道常用命令该怎么用。**

## 当前工作区结构

仓库使用 pnpm 管理工作区依赖，使用 Turborepo 编排构建、测试和 lint 等任务。

当前活跃的业务重点是 CLI 应用本身，因此大部分日常开发会围绕它展开。

## 依赖版本约定

仓库把外部依赖版本集中维护在 workspace catalog 中。

这意味着：

- 外部依赖版本有统一来源
- 各个工作区包不鼓励各自手写分散版本
- 依赖升级时，优先先看全局约定，再落到具体包

对日常协作来说，最重要的是理解这个原则：**版本管理优先集中，而不是分散。**

## 常用命令

```bash
# 安装依赖
pnpm install

# 查看整个 workspace 的过时依赖
pnpm outdated

# 按现有 semver 范围更新依赖
pnpm deps

# 升级到最新版本范围
pnpm deps:latest

# 构建整个仓库
pnpm build

# 仅构建 CLI
pnpm build:cli

# 运行完整验证
pnpm verify
```

## 新增外部依赖时的建议顺序

当你需要引入一个新的外部依赖时，建议按下面顺序操作：

1. 先确认这个依赖是否真的属于仓库公共能力的一部分。
2. 把版本纳入统一的 catalog 管理。
3. 再把它添加到需要使用它的包里。
4. 最后执行安装并更新锁文件。

这样做的好处是，后续升级和排查会更一致，也更容易看清依赖为什么存在。

## 新增工作区内部依赖时的原则

如果未来工作区内的包之间需要互相引用，应保持“明确声明内部依赖”的做法，而不是把它当成普通外部包来处理。

简单说：

- 内部包之间的关系要清楚可见
- 不要让包解析悄悄退回到外部 registry 语义

## 什么时候需要更谨慎

下面几类改动通常值得在提交前做更多确认：

- 调整构建工具链
- 升级大版本依赖
- 修改会影响整个 workspace 的依赖策略
- 修改 CLI 入口或打包相关配置

这类场景下，建议直接执行完整验证，而不是只跑局部命令。
