# 系统总架构

这份文档写给第一次接触 create-yume 的读者，以及准备理解仓库当前结构的人。

如果你读完后只需要完成一件事，那就是：**理解 create-yume 现在由哪些部分组成，以及你接下来该进入哪一组文档。**

## 项目定位

Create Yume 是一个用于创建前端项目的 CLI。

当前它专注在一件事上：帮助你快速生成可直接继续开发的 React 或 Vue 项目骨架，并把仓库已经采用的基础工程约定一并带进去。

## 当前支持范围

目前 create-yume 支持两类项目：

- React 项目
- Vue 项目

它的主路径是：创建一个新项目，并生成一套完整的初始文件。

## 当前不支持的范围

下面这些能力目前不在支持范围内：

- Node 项目脚手架
- 远程模板
- 插件化模板来源
- 对已有项目做增量式改造

如果你的需求落在这些范围内，当前版本并不适合直接使用。

## 当前系统由什么组成

从整体上看，create-yume 目前由五个部分组成：

1. CLI 入口与交互层
2. 配置收集与问题编排层
3. 模板注册、计划生成与渲染执行层
4. 生成后的安装、初始化与验证层
5. 文档与协作约定层

## 当前架构方向

当前仓库不是一个“尚未开始重构”的状态，而是已经进入 ownership-oriented architecture 的后续阶段。

这意味着理解当前系统时，应把下面这些层级视为已存在的现实，而不是未来设想：

1. Preserved Core
2. Scaffold-Family Owner
3. Workspace / Bootstrap Owner
4. Capability Owner

其中，`router` 与 `state-management` 都已经进入 capability owner 的现实状态，不应再按早期试点口径理解系统。

## 架构分层

### 1. CLI 入口与交互层

这一层负责接收命令、处理交互输入，并把不同使用方式统一成同一份项目配置。

它解决的是“用户如何把意图告诉脚手架”。

### 2. 配置收集与问题编排层

这一层负责把交互模式和 preset 模式收束为统一配置，再把配置交给后续生成流程。

它解决的是“生成前需要知道什么”。

### 3. 模板注册、计划生成与渲染执行层

这是脚手架的核心。

它负责：

- 决定在当前配置下需要生成哪些文件
- 决定哪些模板应被渲染
- 组织文件写入、复制和组合型内容生成

它解决的是“最终项目应该被怎样生成出来”。

### 4. 生成后的安装、初始化与验证层

这一层负责项目生成完成后的后续动作，例如安装依赖、初始化协作环境，以及为仓库自身提供基础验证入口。

它解决的是“项目生成后如何落地成一个可继续开发的结果”。

### 5. 文档与协作约定层

这一层负责把项目说明、用户文档、执行约束文档和协作规则分开组织，避免不同读者拿到混杂信息。

它解决的是“不同角色的人，如何进入正确的信息入口”。

## 文档入口如何分流

仓库当前明确区分两类文档：

- 用户文档：服务“了解项目、使用项目、按仓库方式协作”
- 执行文档：服务“修改实现、调整模板系统、处理 Effect 相关实现、选择验证策略”
