# 提交与协作说明

这份文档写给准备向 create-yume 仓库提交代码的人。

如果你读完后只需要完成一件事，那就是：**按照当前仓库约定完成一次可接受的提交。**

## 提交前先做什么

开始提交之前，建议先确认三件事：

1. 依赖已经安装完成。
2. 你的改动范围清楚，知道自己改的是用户层还是实现层。
3. 你准备运行与改动范围相匹配的验证命令。

如果你修改的是内部实现，而不是只改文档，请同时参考执行文档中的验证矩阵。

## 当前提交方式

仓库已经接入了提交辅助工具和基础校验流程。

常用命令如下：

```bash
# 首次配置提交助手
pnpm commit:config

# 交互式生成提交信息
pnpm commit

# 常规 Git 提交
git commit
```

## 提交信息约定

仓库使用 conventional commits。

这意味着提交信息应该能够说明改动类型，例如：

- feat
- fix
- docs
- refactor
- test
- chore

如果你不想手写格式，直接使用 `pnpm commit` 会更稳妥。

## 提交前验证

### 只改文档

以人工校对为主，重点检查：

- 事实是否准确
- 入口是否可发现
- 读者对象是否明确
- 语言是否符合对应文档类型

### 改了代码或模板

建议至少运行：

```bash
pnpm verify
```

如果你明确知道自己只触及局部区域，也可以先按执行文档里的验证矩阵选择最低验证集。

## 什么时候应该看执行文档

如果你碰到下面这些情况，请不要只停留在用户文档：

- 你要修改模板片段或 partial
- 你要调整生成计划逻辑
- 你要改模板引擎能力
- 你要判断某类改动的最低验证范围

这时请进入：

- [执行文档路线图](../agent/roadmap.md)

## 协作建议

为了让后续协作更顺畅，提交时尽量做到：

- 一次提交聚焦一个意图
- 提交信息与改动内容一致
- 文档改动和行为改动保持同步
- 不把用户文档和执行文档混写到一起
